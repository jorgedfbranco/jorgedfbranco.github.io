Introduction

Most modern languages, such as Java, C#, Python, Ruby, Go or JavaScript don’t require direct developer intervention in the management of memory. For the creation of a new Point object, a call to new Point() is enough, and all the particular details of how the respective memory is going to be allocated and eventually deallocated are left to the virtual machine’s garbage collector. Barring Rust and Objective-C, the trend over the last decades is evident: all new languages are garbage collected. We herein define a garbage collection as the process of finding and reclaiming memory known to correspond to objects that are proven to never be used again. Garbage collection per se is just a part of an overall overarching problem – automatic memory management. As a general rule, the garbage collector will need cooperation from the compiler and runtime system to effectively manage the application’s memory.

Heap, Stacks, Metaspace, Native and Code Regions

The developer can directly use two different regions of memory: the stack and the heap (and technically also the native memory area, but that won’t be covered here). Each application thread has an associated stack where local variables, method arguments, return values and return addresses are stored. A stack’s capacity is in general very tight (OpenJDK’s default is of 1 MB for a 64bit process) and stack overflows are the result if an attempt is made to use them beyond that threshold. Stacks are quite inflexible and don’t allow for arbitrary allocation of memory. For that we have the heap, consisting of one or more conceptualluy contiguous regions of memory where objects can be allocated at will. Unlike stacks, heaps tend to be large in size, often having capacities in the order of tens of gigas of memory.

Application Memory Footprint

It’s important to note that the sum of all the running thread’s stack sizes plus heap maximum capacity is not going to correspond to the application’s true memory footprint, as these are not the only process' memory areas. Extra memory areas may contain virtual machine (and garbage collector) data-structures. May contain metadata about class and method definitions. Code on its various forms (bytecode, interpreted or compiled) will also reside in memory. Modern virtual machines will also allow you to use native interfaces that allow you to allocate and manage native memory.

Object Layout

Other than by eventual virtual machine optimizations, in a JVM application any object allocated with the new operator is going to end up allocated in the heap. Knowing for sure how much effective space is being used by a given object is not a simple endeavor. Each object instance will need to have enough space to support all of its fields. It’s customary to prefix each heap object with a 2-word long header containing:

  • Class type (known in HotSpot’s code as “klass word”);
  • GC, lock and hashcode related data (the “mark word”).

On top of that, given padding / alignment constraints, extra space may be kept from being used, so in fact adding to the object’s size.

Garbage Collection Taxonomy

Garbage collectors have existed at least since the early 1960’s, when the first LISP garbage collector was developed by John McCarthy [JMC]. And as with so many other young fields, it took some time for the garbage collection community to converge on a set of widely used definitions – I’ll be using the same vocabulary and definitions as seen in the Garbage Collection Handbook [GCH].

Garbage collection algorithms broadly fall into two main categories. They either free unused memory by reference counting or by tracing.

Reference Counting Collectors

A reference counting collector will keep track, for each object in memory, of the number of inward references from other objects. As new references are made to point to an object, its counter increases, and as those references are removed (by making the source field now point either to another object or to null), the counter decreases. When the counter falls to zero, the object is considered to be unreachable from application code and hence as a target for memory reclamation.

Reference counting exhibits some nice qualities:

  • It has low memory overhead. Objects just need to have a counter that for most cases doesn’t even need to have that many bits. That can be most likely accomodated by the commonly seen 2-3 words object headers. Unlike tracing algorithms like the copy collector, no extra memory is needed to be had on the side, just as an operational cost;
  • Deallocation can happen right at the moment the object is deemed unreachable. This contrasts drastically with tracing algorithms that have to wait for a mark phase for the identification of objects no longer alive. This could mean that, at least apparently, no long pauses are required by the garbage collector, unlike most tracing algorithms. We will see that this ends up not being the case;
  • It’s a non-moving collector. Objects are not moved in memory. This allows reference counting to be used not only in type-safe languages, but also in languages like C and Rust. In fact, manual versions of reference counting are quite common in unsafe languages. See conservative collector.

But also has very substancial drawbacks:

  • The algorithm is unable to process cyclical references. Consider the case where an object A points to an object B and vice-versa. Let’s also consider that object A is being referenced by some other reachable object X in the heap. That would translate into refCount(A) = 2 and refCount(B) = 1. One would expect that the moment that X drops the reference to A, both A and B could be reclaimed, but in fact that will only decrease refCount(A) from 2 to 1. A and B will never be reclaimed! In practice, for the virtual machine to behave correctly even in the presence of cyclic references, a fallback tracing GC needs to be run from time to time;

  • Counters may at times overflow. When this happens, and depending on the algorithm implementation, objects can be kept in memory forever. The usual solution is to resort to a fallback tracing collector, a system that would anyway have to be in place to deal with cyclic references;

  • Even read-only operations can entail a huge amount of heap mutations. Consider the case where one iterates over an arbitrary array, with the sole intent of reading its contents. As we move our pointer marking the current element through the array, at each step at index A[i] we’ll need to decrease the count of element A[i] as well as increasing the count of A[i]. This high level of heap mutations even in read-only operations can be quite troublesome in a highly concurrent setting, as a great deal of modern applications are;

  • Pause times can be quite unpredictable. Although at first glance it may look like no long pauses can happen with reference counting, in practice they can still happen, and often very unpredictably. Consider the case of a million node single-linked list with no external object pointing to any of its nodes, other than object X having a single reference to the list’s head node. The moment this reference is cut off, the immediate effect will be of setting refCount(head) = 0. But before the marking of the node’s memory as ready for collection, all the node’s fields need to be scanned and any found references further processed. The next node in the chain will also see its refCount(node) turn to zero, generating a cascade effect all throughout the list, one million times.

Tracing Collectors

Tracing collectors take a completely different approach to garbage collection. They divide the collection stage in two phases – a marking phase, where all reachable objects are identified and marked as such, followed by a phase tasked with the reclamation of available memory proper. As it turns out, there are only so many ways you can gargabe collect with tracing. The three possible tracing garbage collection variants are: Copying collection, Compacting collection and Sweeping collection. All the buzzwordy new GCs you have heard about will in way or another be a variant or combination of these three.

The Marking Phase

All tracing collectors have as a first phase the identification and marking of all live objects. Marking (tracing) starts by adding to the mark-stack all GC roots, that is, all static fields or any kind of pointer emanating from application stacks (this includes local variables and method arguments). The mark-stack is then processed until empty: each object popped from the stack will get marked as alive and any references found in its fields that’s still not marked as alive will be pushed into the stack. By the time the stack is empty we know that any object not marked must be unreachable, and hence, a target for collection.

We’ll now go over the different kinds of tracing collectors, and see how their subsequent phases work.

Mark / Copy Collector

Mark / Compact collectors divide the heap into two areas, called from and to. Objects are always allocated in the from area. When the from area is full, a marking phase is kicked off. Subsequently, all the alive objects in the from area are copied into the to area. The final step is to flip the to and from areas: the to area is now the from area and vice-versa. The garbage collection process is by then over.

The copy collector always requires heap sizes to be twice as large than the real application needs, as objects can only be allocated in one of the halves at a time. On the other hand, the algorithm will only mark and copy live objects – the work to be performed is proportional to the live-set size and not to the heap size, which works well with workloads with huge mortality rates. Also, after copying all the objects will be nicely packed (compacted) next to each other, which is good both for cache-friendiness and for fragmentation.

Mark / Compacting Collector

A Mark / Compact collector will, after the marking phase, compact all objects into one of the ends of the heap. There are multiple techniques to do so. A good compactor should make sure to “slide” the objects without changing their order, as that tends to yield the best results in terms of cache-friendliness.

The drawbacks of compaction are highly dependent on the actual algorithm used. Most of the approaches require multiple passes (Java’s Concurrent Mark & Sweep fallback compactor needs 4 passes, for instance. Shenandoah [SHE] works similarly [SHEC]), may involve the reversal of pointer directions or even the existence of an external forwarding pointer in the header of all objects (the LISP2 approach [LISP2]).

Garbage Collector Properties

Mutator Threads vs Collector Threads

Applications running in modern CPUs and operating systems support one or more running threads. Threads can come in two types:

  • Collector Threads, threads that don’t run any kind of application code and are thus only used for garbage collection. These threads don’t necessarily run at all times, and will only kick in when requested by the virtual machine;
  • Application Threads, frequently also referred to as mutator threads. These are the threads created and directly or indirectly managed by the developer of the application, as they’re tasked with running the (business) application code. They are considered to be mutators, as from a garbage collection perspective, they actively mutate the object graph in the heap.

Concurrent Collectors

A garbage collector is said to be concurrent if collector threads run, even if only for some stages, simultaneously with application code.

If a garbage collector is non-concurrent, garbage collection events force the pause of all application threads for the full duration of the collection. The stopping of all application threads to perform virtual machine house-keeping is called entering a safepoint. In the context of garbage collection, the expression Stop The World (STW) is also commonplace.

As most garbage collectors execution time is proportional to the heap size or at least to the live-set size, as heaps grow larger and larger by the year, STW events are getting more and more pronounced. If we’re to assume that a 500 MB heap can be processed in half a second, then a 16 GB heap will then take some 8 seconds to be processed. If the application is frozen throughout the whole process it will be undoubtly noticed by the user (if it’s a UI we’re dealing with) or by external applications (that may be assuming requests to be returned promptly, or heartbeats to be regularly sent at the very same short time intervals).

Parallel Collectors

A garbage collector is said to be parallel if there is at any time more than one collector thread running. A collector that is able to execute the mark phase of a mark & sweep collection with multiple threads is an example of a parallel collector. Beware that this property is orthogonal to whether or not there are mutator threads currently running or not. A collector may be parallel during a stop the world event, or may be fully concurrent with mutator threads but run with a single collector thread (so, not parallel).

Compacting Collectors

A nice property of a garbage collector is in its ability to compact memory as it goes. The reasons are twofold:

  • Objects tightly stacked together tend to play nicer with CPU caches. The more spread they are throughout the heap, the more caches misses will be incurred by the application when attempting to access them. In modern architectures the CPU processing speed is so much faster than memory access latencies that up to there levels of cache sit between the CPU and main memory to attempt to minimize those roundtrips as much as possible. Thus, having an application displaying cache friendlyness is of vital importance.
  • Less (external) fragmentation. An application that never compacts its objects will tend to form “holes” in the heap. The more and smaller those “holes” are, the harder it will become to allocate new objects in memory – the application will take more and more time and processing power trying to find a “hole” large enough to fit the allocation request. This can be particularly problematic when the application attempts to allocate large objects. It can be quite puzzling at times to see that the application seems to be running out of memory although plenty of heap still seems to be available. Reference-counters and Mark & Sweep collectors are non-compacting collectors and they are known to suffer from fragmentation issues. In commercial-grade VMs, the situation tends to be ameliorated by having a compacting collector run as a last resort before throwing out an OutOfMemoryError.

Regionalized Collectors

Regionalized collectors divide the heap into multiple, equally sized memory regions. An example would be the G1 GC [G1GC], where the heap is divided in 2048 regions, with each region not being larger than 2MB. Other GCs may consider smaller regions, instead. The Compressor GC [COMP], for instance, has its region be the size of an operating system page.

Beware that generational collectors may also divide the heap in multiple memory areas, but that’s not what one usually means by a regionalized GC.

Generational Collectors

It didn’t take too long for two ideas being noticed:

  • Most objects die young. It’s quite common in server applications to see survival rates of recently allocated objects in the order of 3-5%;
  • When objects do age, they tend to stay alive in memory for long periods.

Given these two observations, it turns out that running a full garbage collection over the whole heap every time new memory needs to be reclaimed may be inefficient.

Incremental Collectors

At times it may happen that all the collecting work is performed by collector threads. At other times, application threads may be asked to help with the garbage collecting process. A collector is said to be incremental if at least part of the collection work is performed by application threads. Two common examples are:

  • Mark & Sweep: The sweeping phase may not actually be conducted by collector threads, but instead, it may be executed lazily by application threads only when attempting to allocate memory and none is available;
  • Concurrent, Compacting collectors: Certain concurrent, compacting collectors will run in two phases: first they mark all the live objects and then decide which regions need to be evacuated. All the live objects in a region marked to be evacuated will be moved to a new region and when the process is finished the original region can be marked as free for new allocations. It may happen that thousands of regions are marked as needing evacuation but it may take some time for the collector threads to process them all. Conservative Collector

A conservative collector runs in unsafe languages, such as C or C++. Such a collector must be way more conservative in its assumptions than the typical Java or C# collector. In one hand, and given that the developer is in direct control of memory, it needs to be very careful in the way it interprets memory. On the other hand, it doesn’t have, for the most part, any kind of collaboration of the runtime system: no runtime type information or reflection capabilities like the typical, modern, safe language does.

What this translates to, in practical terms, is to having a garbage collector that cannot under any circumstances move objects around (hence, a non-moving collector), as that would break any reference pointing to the moved object, and it may need at times to treat arbitrary integers in memory as pointers, as it may be unable to ascertain with total confidence whether a given value in memory is a number or value or a pointer proper. This of course makes the algorithm have to work a lot harder while yielding inferior results when comparing to its non-conservative counterparts.

Conclusion

We have by now surveyed the major topics in the garbage collection landscape.
