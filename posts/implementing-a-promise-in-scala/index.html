Implementing a Promise in Scala

March 17, 2020

Roadmap: We describe promises in Scala and how they can be used. As a theoretical recap, we see how to use the wait() / notify() pattern with intrinsic locks, and how to implement the same algorithms on top of ReentrantLocks. With this, we have all the necessary artillery to do a simple promise implementation. This implementation will not be a drop-in replacement in Scala, we'll see why and compare it with Scala's Promise[T].

/**
 * A thread-safe container for a value that can be set by calls to `set()`
 * and retrieved by calls to `get()`. Any of those methods can be invoked
 * by any thread. If a call to `get()` is performed before any value was
 * assigned via a `set()` call, the invocation blocks until a call to `set()`
 * is performed by another thread.
 */
trait Promise[T] {

  /** 
   * Returns the value defined by a call to `set()`. 
   * If at the time of the call no value was set on this promise, the calling
   * thread blocks until one is available.
   */
  def get(): T
  
  /**
   * Allows you to define a callback `f` to be invoked as soon as the promise
   * promise has a value. Beware that `f` should execute quickly -- this
   * implementation will execute all callbacks on the thread executing the
   * `set()` call!
   */
  def onComplete(f: => Unit): Unit
  
  /**
   * Sets a value to this promise. 
   * Can only be called once; subsequent calls will result in an exception
   * being thrown.
   */
  def set(value: T): Unit
}

Before delving into the promise implementation per se, some JVM preliminaries are needed.

A typical use-case for the usage of a promise is having a method f creating a promise p and then handing it off to some callback method, and have the callback eventually set its value. One may then schedule some code to be run when the promise is completed:

def f(..., callback: Promise[String] => Unit) = {
    ...
    val p = new Promise[String]()
    callback(..., p)
    p.onComplete { doSomething() }
}

Theoretical Foundations

Java’s java.util.concurrent provides a set of classes covering a lot of the common of use-cases faced when developing concurrent applications on the JVM – take Semaphore, CyclicBarrier, ConcurrentHashMap, AtomicReference or BlockingQueue as examples. Out of interest or academic curiousity, one may wonder how to implement those very same data-structures from scratch, while only using Java’s concurrency primitives. Is that even possible to do or do they are partly implemented by use of native code? As with most things in life, the answer is nuanced. Most of the logic is standard Java code, but a large part of it resorts to a backdoor class named sun.misc.Unsafe, providing access to low-level native operations. The misc.Unsafe class was never intended to be used by the general public, but mostly for performance reasons ended up being extensively used by library writers or developers of highly concurrent applications where speed matters. Other than that, yes, one should be fine implementing java.util.concurrent's data-structures while resorting only to Java’s constructs.

As a motivating example, let’s consider the following example: we’re developing a blocking queue and would like our poll() method to return the oldest element in the queue if such element exists, or to block the calling thread until one is available. How can we express such an algorithm just by resorting to the synchronize keyword and by using volatile variables? The answer is that a performant solution is just not possible. But with the help of three methods that are readily available in all Java objects, we will. They are wait(), notify() and notifyAll().

Implementing a Promise[T]

With the given theoretical background things get quite simple. A Promise[T] is can

case class PromiseImpl[T]() {
  @volatile private var value: T = _

  // CAREFUL: also works as a synchronization target!
  private val listeners = new ArrayBuffer[T => Unit]()

  def get(): T = {
    listeners.synchronized {
      while (value == null)
        listeners.wait()
      value
    }
  }

  def onComplete(f: T => Unit): Unit = {
    listeners.synchronized {
      if (value == null) {
        listeners += f
      } else {
        f(value)
      }
    }
  }

  def set(v: T): Unit = {
    listeners.synchronized {
      if (value != null)
        sys.error("Promise value was already previously set.")
      value = v
      listeners.notifyAll() // unblock all `get()` calls
      listeners.foreach(_(v)) // invoke all `onComplete` callbacks
    }
  }
}

A cleaner approach can be had by using an AtomicReference to keep the promise’s value, instead of our mutable volatile variable:

case class Promise[T]() {
  private val value = new AtomicReference[T]()

  def get(): T = {
    value.synchronized {
      while (value.get() == null)
        value.wait()
      value.get()
    }
  }

  def set(v: T): Unit = {
    value.synchronized {
      if (value.get() != null)
        sys.error("Promise value was already previously set.")
      value.set(v)
      value.notifyAll()
    }
  }
}

Scala’s Implementation

Scala’s current promise implementation (version 2.13) is substantially different than the one here described. In Scala, a promise only directly allows you to set the value of the promise, but to consume it you’ll always have to use its future: Future[T] method.

Both promises and futures allow you to perform a blocking get() operation, and from the consumer side, there’s no difference between a promise and a future. Thus, it comes as no surprise that Scala’s promise implementation contains an extra method we did not include, future: Future[T], that allows you to get a future out of your promise.

The goal of the article was to describe a simple implementation of a promise. We did not cover futures and how they relate to promises. Scala, on the other hand, supports both promises and futures, and it turns out that DefaultPromise[T], the backing class of scala.concurrent.Promise[T], is also a future.
