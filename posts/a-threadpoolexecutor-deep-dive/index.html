A ThreadPoolExecutor Deep-Dive

March 19, 2020

Roadmap: We provide a motivation for the introduction of the ThreadPoolExecutor and Executors classes in JDK 1.5. We describe how to create fixed and cached thread-pools through Executors's factory methods, and then how to create thread-pools by manual instantiation of the ThreadPoolExecutor class. We finally proceed to dig into the implementation of ThreadPoolExecutor, raising warnings on the behaviors experienced by uncareful selection of instatiation parameters.

Introduction

Java has been designed from its inception with multi-threading in mind. Unfortunately, and although all the needed concurrency primitives were from the very beginning made available to developers, development could quickly get unwildy: a lot of boilerplate code had to be recreated time and time again across applications, and that was quite often not even the biggest hurdle – it’s no secret that getting concurrent algorithms right is hard.

To aid in that regard, Java 1.5 introduced a java.util.concurrent package containing a wealth of concurrency utility classes and algorithms. Among them, the ThreadPoolExecutor class was brought about, unshackling developers from the need to manually create, manage and dispose of threads in their multi-threaded programs. It was now enough to set up one or more ThreadPoolExecutor instances; all then left to do was to submit tasks (in the form of Runnable / Callable[T] instances) to then be asynchronously executed on your behalf by the thread-pool:

Theoretical Background - A Recap

We’ll summarize the main interacting classes with ThreadPoolExecutor: Runnable, Callable[T], Future[T], and ThreadFactory:

Runnable / Callable[T]

Two different kind of tasks can be submitted to a thread-pool. Tasks with no return value, defined through Runnable instances:

threadPool.submit(new Runnable {
  override def run(): Unit = {
    ...code to execute
  }
})

and tasks with a return value, defined through Callable[T] instances:

threadPool.submit(new Callable[Int] {
  override def call(): Unit = {
    ...code to execute
    666
  }
})

Future[T]

When submitting a task to a thread-pool, a Future[T] instance is returned. Submitted runnables yield a Future[Unit], while callables of type T yield a Future[_] (Future<Void>, in Java).

val fx: Future[_] = threadPool.submit(new Runnable { ... })
val fy: Future[Int] = threadPool.submit(new Callable[Int] { ... })

Calls to Future.get() will either immediately return the evaluated result, if there already is one, or block the calling thread until the task is completed. If the task threw an exception, the call to Future.get() will then also throw that exception. Invoking Future.get() multiple times is an idempotent operation.

ThreadFactory

When a ThreadPoolExecutor needs to create a thread, it makes a call to ThreadFactory.newThread(Runnable):

public interface ThreadFactory {
    Thread newThread(Runnable runnable);
}

As ThreadPoolExecutor constructors allow you to pass your own ThreadFactory implementation, thread creation can be controlled by the client. The most common fields to set are:

override def newThread(runnable: Runnable): Thread = {
  val thread = new Thread()
  thread.setName(...) // give each thread its own, recognizable, name.
  thread.setDaemon(...) // set the thread as a daemon (background) thread.
  thread
}

Naming threads

A thread-pool most often will have multiple threads, so their names tend to have the same prefix, plus a counter. So, if running a CPU-bound thread-pool and a IO-bound thread-pool, a thread-dump could show you threads such as:

  • …
  • io-bound-thread-pool-0
  • io-bound-thread-pool-1
  • …
  • cpu-bound-thread-pool-0
  • cpu-bound-thread-pool-1
  • …

Daemon threads

When attempting to monitor or diagnose runtime problems, you’ll want to be able to easily identify what thread is running what. Do note that any string can go into the thread name, and nothing is going to prevent you to assign the very same name to multiple threads – they’re there to assist the developers, and are never used as internal ids of any sort.

When a process’ main() method returns, the application will still stay running as long as there’s at least one other foreground thread running. The moment all foreground threads die, the process is finally shut down. All threads are by default created as foreground threads, but it’s common for you to not want to keep the application running just because you have a thread-pool running. It may even be that the thread-pool isn’t even currently processing any task! In that case, you can mark your threads as daemon threads (via setDaemon(true)) – this will make sure your process shuts down the moment your main() method returns.

With this quick recap concluded, we’re free to move on and see how to use thread-pools in Java.

Thread-Pool Usage

public interface Executor {
    void execute(Runnable command);
}

public interface ExecutorService extends Executor {
    <T> Future<T> submit(Callable<T> task);
    <T> Future<T> submit(Runnable task, T result);
    Future<?> submit(Runnable task);

    void shutdown();
    List<Runnable> shutdownNow();
    boolean isShutdown();
    boolean isTerminated();
    boolean awaitTermination(long timeout, TimeUnit unit) throws InterruptedException;

    ...
}

The ThreadPoolExecutor constructors take multiple parameters, and as we will soon discover, setting them up in a way that makes the thread-pool behave in predictable ways has its nuances. Thus, the generally recommended way of instantiating thread-pools is to make use of the Executors singleton. It provides a rich set of factory methods that make thread-pool instantiation a breeze. We will focus only on two of such cases: fixed thread-pools and cached thread-pools:

val cpuCount = Runtime.getRuntime().availableProcessors()
val threadPool = Executors.newFixedThreadPool(cpuCount)

Further Notes

ScheduledThreadPoolExecutor

Java 1.5 also brought another kind of thread-pool, directly extending ThreadPoolExecutor: the ScheduledThreadPoolExecutor, allowing for the scheduling of tasks to be run at an arbitrary time in the future.

The ScheduledThreadPoolExecutor works by having a main driving thread that controls the scheduling logic (deciding when to trigger the submitted tasks). This thread lives outside the thread-pool. The thread-pool threads are only used when the time has come to run a given task: the driving thread will at that moment submit the task to the thread-pool and it will eventually executed.

ForkJoinPool

Java 1.7 introduced the new ForkJoinPool class, a new type of thread-pool based on task workstealing. Although its name doesn’t give it away, it’s also an ExecutorService, just like ThreadPoolExecutor is. New factory methods were added to the Executors singleton to make its instantiation simpler.

Conclusion

The ThreadPoolExecutor is a simple and effective thread-pool, particularly shinning when running longer tasks. Lots of short tasks, or tasks spawning other tasks, can be problematic, as every task submission to the thread-pool and processing by a thread involves synchronization logic.
