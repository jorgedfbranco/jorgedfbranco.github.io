A ThreadPoolExecutor Deep-Dive

March 19, 2020

Roadmap: We provide a motivation for the introduction of the ThreadPoolExecutor and Executors classes in JDK 1.5. We describe how to create fixed and cached thread-pools through Executors's factory methods, and then how to create thread-pools by manual instantiation of the ThreadPoolExecutor class. We finally proceed to dig into the implementation of ThreadPoolExecutor, raising warnings on the behaviors experienced by uncareful selection of instatiation parameters.

Introduction

Java has been designed from its inception with multi-threading in mind. Unfortunately, and although all the needed concurrency primitives were from the very beginning made available to programmers, development could quickly get unwildy: a lot of boilerplate code had to be recreated time and time again across applications, and that was quite often not even the biggest hurdle, as it’s no secret that getting concurrent algorithms right is hard and even once implemented, hard to verify for correctness.

To aid in that regard, Java 1.5 introduced a java.util.concurrent package containing a wealth of concurrency utility classes and algorithms. Among them, the ThreadPoolExecutor class was brought about, unshackling developers from the need to manually create, manage and dispose of threads in their multi-threaded programs. It was now enough to set up one or more ThreadPoolExecutor instances; all then left to do was to submit tasks (in the form of Runnable / Callable[T] instances) that would then be asynchronously executed on your behalf by the thread-pool:

val threadPool = new ThreadPoolExecutor(...)
threadPool.submit(() => println("Hello World!"))

Let’s now take a more careful look on how to use a ThreadPoolExecutor.

Thread-Pool Instantiation

ThreadPoolExecutor provides multiple constructor variants to pick from. Unfortunately, they are all quite bulky and require some understanding of the internal workings of the ThreadPoolExecutor class to get right. To facilitate thread-pool instantiation, an Executors singleton class consisting of factory methods was provided, but not all of them are ThreadPoolExecutor related. The only (overloaded) façade methods for ThreadPoolExecutor are:

  • ThreadPoolExecutor.newFixedThreadPool(threadCount)
    ; 
  • ThreadPoolExecutor.newFixedThreadPool(threadCount, threadFactory)
    ;

and

  • ThreadPoolExecutor.newCachedThreadPool()
    , 
  • ThreadPoolExecutor.newCachedThreadPool(threadFactory)
    ,

which will now be described.

Fixed Thread-Pools

A fixed thread-pool keeps the number of threads it’s running fixed throughout its lifetime. These thread-pools are quite often used for CPU-bound tasks, being assigned as many threads as the physical core count – after all, there’s no point in having more threads running than there are physical CPUs. If the available threads are not able to keep up with the influx of submitted tasks, tasks will get queued and will be, when possible, processed “first in, first out”.

Fixed thread-pools can be instantiated through the Executors singleton:

val cpuCount = Runtime.getRuntime().availableProcessors()
val threadPool = Executors.newFixedThreadPool(cpuCount)

Cached Thread-Pools

A cached thread-pool dynamically adjusts its number of running threads. It attempts to reuse idle threads as tasks get submitted, but has no qualms in adding new threads to guarantee all submitted tasks get executed on-the-spot. Given this, a task is guaranteed to never queue, unlike with fixed thread-pools. Threads have a keepAliveTime, and any thread seen to stay idle for greater periods is killed. The “cached” in cached thread-pools most likely stems from its keeping of threads alive for a given period of time, allowing for some thread reuse (caching) throughout time.

Cached thread-pools can be instantiated as such:

val threadPool = Executors.newCachedThreadPool()

Task Submission

Tasks can be submitted by invoking:

  • ThreadPoolExecutor.execute(task: Runnable)
    ; 
  • ThreadPoolExecutor.submit(task : Runnable): Future[_]
    ; 
  • ThreadPoolExecutor.submit[T](task: Callable[T]): Future[T]
    .

Remember the difference between Runnable and Callable[T]: both represent a task to be run, but while the former doesn’t return a value, the latter returns a value of type T.

Do notice that calls to submit() return futures: runnables yield a Future[_] (Future<Void>, in Java), while callables of T yield a Future[T].

val fx: Future[_] = threadPool.submit(new Runnable { ... })
val fy: Future[Int] = threadPool.submit(new Callable[Int] { ... })

Calls to Future.get() will either immediately return the evaluated result, if there already is one, or block the calling thread until the task is completed. If the task threw an exception, the call to Future.get() will then also throw that exception. Future.get() is an idempotent operation.

Thread-Pool Shutdown

Thread-pools may at any time be stopped. To do so, one may call:

  • shutdown(), that will remove and ignore any tasks currently sitting in its internal queue and will ignore any new task submissions. Currently executing tasks are left unharmed and will naturally run;

  • shutdownNow(), behaving similarly to shutdown(), but that will also try to stop execution of already running tasks, via calls to Thread.interrupt(). Beware that calls to Thread.interrupt() do not guarantee thread termination – they will set a thread’s interrupt flag and its up to the task’s logic to constantly poll for the thread’s interrupt status flag (through Thread.currentThread().isInterrupted())) and stop executing if it’s set. In other words, it works by convention. Given that thread interrupts are not a very well known topic, this renders shutdownNow() very similar in practical terms to shutdown().

Beware that both these methods work asynchronously. If you want to know block until the thread-pool is finished and its threads and queues fully disposed, you can invoke awaitTermination(). If you just need to know whether the thread-pool got terminated (shutdown), you call call isTerminated().

ThreadPoolExecutor Implementation

The ThreadPoolExecutor works by internally keeping a blocking queue to which tasks are submitted:

// pseudo code

def submit(callable: Callable[T]): Future[T] = {
  val promise = Promise[T]()
  this.tasks.offer { () =>
    promise.completeWith(Try(callable.run()))
  }
  promise.future
}

while on the consumer side, all threads are constantly either executing a task or awaiting for a new task to be available in the queue:

// pseudo code

while (!this.isTerminated) {
  val task = this.tasks.poll() // blocking call
  task.run()
}

From a performance perspective, this implies synchronization costs both when submitting the task to the blocking queue and when polling out of it. As long as each task contains quite some work to do, that overhead may be safely ignored, but one should be careful to not overflow the thread-pool with zillions of small tasks, as the blocking queue may then turn into a bottleneck.

Constructor Parameters

Although ThreadPoolExecutor provides multiple constructors, they are all contain the same core parameters:

  • corePoolSize - The number of threads we want to keep around at all times, even if the threads have been idle for a while;
  • maximumPoolSize - The maximum number of threads we want our thread-pool to simultaneously have at any point in time;
  • keepAliveTime, unit - The maximum time we are willing to allow an idle thread to exist before it gets killed;
  • workQueue - The blocking queue this thread-pool should use keep its tasks.

Further Notes

ScheduledThreadPoolExecutor

Java 1.5 also brought another kind of thread-pool, directly extending ThreadPoolExecutor: the ScheduledThreadPoolExecutor, allowing for the scheduling of tasks to be run at an arbitrary time in the future.

The ScheduledThreadPoolExecutor works by having a main driving thread that controls the scheduling logic (deciding when to trigger the submitted tasks). This thread lives outside the thread-pool. The thread-pool threads are only used when the time has come to run a given task: the driving thread will at that moment submit the task to the thread-pool and it will eventually executed.

ForkJoinPool

Java 1.7 introduced the new ForkJoinPool class, a new type of thread-pool based on task workstealing. Although its name doesn’t give it away, it’s also an ExecutorService, just like ThreadPoolExecutor is. New factory methods were added to the Executors singleton to make its instantiation simpler.

Conclusion

The ThreadPoolExecutor is a simple and effective thread-pool, particularly shinning when running longer tasks. Lots of short tasks, or tasks spawning other tasks, can be problematic, as every task submission to the thread-pool and processing by a thread involves synchronization logic.
